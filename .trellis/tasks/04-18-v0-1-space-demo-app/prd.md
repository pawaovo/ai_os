# Build V0.1 Space Demo App

## Goal

Create the first locally runnable AI Space demo app so the product can be opened in a browser and manually tested.

This is not the final desktop app. It is a V0.1 validation surface for the core loop:

```text
Goal input -> Companion -> Control Plane -> Executor Adapter -> Run/Event status -> Artifact output -> Space shell
```

## Requirements

- Add a local dev command that starts a browser-accessible app from `product/`.
- Reuse the existing `apps/space-desktop` package rather than creating a separate product surface.
- Show a simple goal/chat input.
- Show Mission and Run status.
- Show stream output and Artifact output.
- Provide a deterministic mock executor path for reliable local manual testing.
- Keep Codex and Claude Code as visible executor targets only if the UI can represent them without forcing real API/process usage.
- Preserve current boundaries: Companion does not call executors directly; Control Plane owns Mission/Run orchestration; executor adapters normalize events.

## Acceptance Criteria

- [x] `cd product && npm install` works with the committed lockfile.
- [x] `cd product && npm run dev` starts a local app.
- [x] The app can be opened in a browser and displays the V0.1 Space demo shell.
- [x] Entering a goal and running the demo shows status moving through running/completed.
- [x] The demo displays at least one artifact generated by the run.
- [x] The implementation has automated tests for the demo runtime and app shell behavior.
- [x] `cd product && npm test` passes.

## Non-Goals

- No database.
- No account system.
- No cloud runtime.
- No real desktop packaging.
- No polished infinite canvas.
- No Forge shell.
- No store/marketplace.
- No complex automation scheduler.
- No real external model call required for the first visible demo.

## Technical Notes

- Keep the implementation small and browser-first.
- Prefer a lightweight dev server/build path over a full desktop framework.
- If a bundler is needed, use it only to make the demo locally runnable; do not introduce a UI framework unless the current code requires it.
- The mock executor should emit normalized `KernelEvent` through the existing `CodeExecutor` protocol.
